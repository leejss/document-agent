# PRD: Planner-Executor 기반 문서 작성 에이전트

## 1. 문서 개요

### 제품명

Planner-Executor 기반 문서 작성 에이전트

### 문서 목적

단일 LLM 호출 방식의 한계를 해결하기 위해, 문서 작성 과정을 계획과 실행으로 분리하는 문서 작성 에이전트의 요구사항을 정의한다.

### 배경

기존의 일반적인 LLM 문서 생성 방식은 한 번의 프롬프트로 전체 문서를 작성하려고 시도한다. 이 방식은 간단한 요청에는 빠르게 대응할 수 있지만, 길고 구조적인 문서에서는 다음과 같은 문제가 반복적으로 발생한다.

* 문서 전체 길이가 짧다
* 섹션별 설명 깊이가 부족하다
* 아웃라인 품질이 불안정하다
* 핵심 섹션이 누락되거나 중복된다
* 섹션 간 논리 흐름이 약하다
* 문체와 용어가 일관되지 않다

문서 작성은 본질적으로 단일 생성이 아니라 다음과 같은 다단계 작업이다.

* 목적과 독자 파악
* 문서 구조 설계
* 섹션별 초안 작성
* 예시와 세부 설명 보강
* 전체 문서 편집 및 검토

따라서 문서 작성을 하나의 생성 호출이 아니라, 계획과 실행이 분리된 워크플로우로 재구성할 필요가 있다.

---

## 2. 문제 정의

### 현재 문제

사용자는 기술 문서, 가이드, 제안서, 설계 문서처럼 구조가 중요한 긴 문서를 생성하려고 할 때 아래 문제를 겪는다.

1. **아웃라인 생성 실패**

   * 문서의 큰 흐름이 제대로 설계되지 않는다
   * 상위/하위 목차 구조가 불안정하다
   * 필수 섹션이 빠지거나 순서가 어색하다

2. **본문 길이와 깊이 부족**

   * 전체 분량이 기대보다 짧다
   * 각 섹션이 설명 대신 요약으로 끝난다
   * 예시, 비교, 근거, 코드가 빠진다

3. **문서 일관성 부족**

   * 섹션마다 용어와 톤이 달라진다
   * 같은 설명이 여러 섹션에서 중복된다
   * 서론, 본문, 결론이 자연스럽게 연결되지 않는다

4. **확장성과 제어성 부족**

   * 특정 섹션만 재작성하거나 보강하기 어렵다
   * 작성 과정을 추적하기 어렵다
   * 문서 품질 개선 포인트를 시스템적으로 반영하기 어렵다

### 근본 원인

* 문서 작성 전체를 한 번의 LLM 호출에 과도하게 의존함
* 구조 설계와 본문 생성을 분리하지 않음
* 섹션 단위 목표, 분량, 중복 방지 규칙이 없음
* 전역 편집과 품질 검토 단계가 없음

---

## 3. 제품 목표

### 핵심 목표

Planner-Executor 패턴을 사용하여, 긴 문서에서도 안정적인 아웃라인과 충분한 분량, 일관된 품질을 제공하는 문서 작성 에이전트를 만든다.

### 세부 목표

1. 문서 목적과 독자를 반영한 고품질 아웃라인 생성
2. 섹션별 충분한 길이와 설명 깊이 확보
3. 병렬 작성이 가능한 구간에서 속도와 생산성 향상
4. 섹션 간 중복 감소 및 용어 일관성 확보
5. 최종 편집 및 리뷰를 통해 완성도 높은 문서 생성
6. 특정 섹션 재작성과 보강이 가능한 구조 제공

### 비목표

초기 버전에서 아래는 포함하지 않는다.

* 실시간 협업 편집
* 외부 검색 기반 근거 자동 인용
* 멀티모달 자료 해석
* 사람 승인 기반 단계별 워크플로우
* 복잡한 지식 그래프 추론
* 팀 단위 버전 관리 및 승인 프로세스

---

## 4. 타겟 사용자

### 1차 사용자

* 기술 문서를 자주 작성하는 개발자
* PRD, 가이드, 설계 문서를 작성하는 PM
* 설계 제안서와 내부 문서를 작성하는 플랫폼/백엔드 엔지니어

### 사용자 특성

* 문서를 자주 쓰지만 구조 설계와 분량 확보에 시간을 많이 씀
* 단일 LLM 호출 결과가 “짧고 얕다”는 불만이 큼
* 문서의 완성도뿐 아니라 수정 용이성도 중요하게 생각함

## 5. 주요 사용 시나리오

### 시나리오 1: 기술 설계 문서 작성

사용자는 “planner-executor 문서 작성 에이전트 설계”를 주제로 기술 설계 문서를 작성하고 싶다. 시스템은 독자와 목적을 분석한 뒤 아웃라인을 설계하고, 섹션별 초안을 작성하고, 최종적으로 일관된 설계 문서를 생성한다.

### 시나리오 2: 긴 가이드 문서 작성

사용자는 특정 기술 주제에 대한 가이드 문서를 길고 자세하게 작성하고 싶다. 시스템은 섹션별 최소 분량을 보장하며 초안을 생성하고, 누락된 예시와 보충 설명을 추가한다.

### 시나리오 3: 특정 섹션만 보강

사용자는 생성된 문서 중 “병렬 실행 전략” 섹션이 약하다고 판단한다. 시스템은 전체 문서를 다시 쓰지 않고 해당 섹션만 재작성하거나 확장한다.

### 시나리오 4: 여러 섹션 동시 초안 생성

아웃라인이 확정된 뒤, 서로 독립적인 섹션들을 병렬로 작성하여 생성 시간을 줄인다. 이후 editor가 전체 문서를 하나로 정리한다.

## 6. 제품 원칙

1. **문서 작성은 한 번에 끝내지 않는다**
   계획, 작성, 편집, 검토를 분리한다.

2. **아웃라인은 전역적으로 설계한다**
   최초 구조 설계는 단일 planner가 책임진다.

3. **본문은 섹션 단위로 생성한다**
   각 섹션은 독립적인 목표와 분량 기준을 가진다.

4. **병렬 실행은 제한적으로 적용한다**
   섹션 초안 작성에는 병렬을 쓰되, 최종 구조 설계와 편집은 직렬로 유지한다.

5. **최종 품질은 editor와 reviewer가 책임진다**
   생성만으로 끝내지 않고, 통합과 검토를 통해 문서 완성도를 높인다.

## 7. 솔루션 개요

### 아키텍처 요약

문서 작성 에이전트는 다음 단계로 동작한다.

1. Request Analyzer
2. Planner
3. Section Executor Pool
4. Intro/Conclusion Writer
5. Editor
6. Reviewer
7. Patch/Rewriter

### 고수준 플로우

```text
User Request
  → Request Analyzer
  → Planner
  → Section Briefs 생성
  → 병렬 가능한 섹션들 동시 작성
  → 의존성 있는 섹션 순차 작성
  → 서론/결론 작성
  → 문서 병합
  → Editor
  → Reviewer
  → 약한 섹션 재작성
  → Final Document
```

## 8. 기능 요구사항

## 8.1 입력 처리

시스템은 최소한 다음 입력을 지원해야 한다.

* 문서 주제
* 문서 목적
* 대상 독자
* 문서 형식
* 톤
* 목표 길이
* 반드시 포함할 항목
* 제약조건

### 기능 요구사항

* 사용자가 최소 입력만 제공해도 기본 문서 생성을 시작할 수 있어야 한다
* 입력이 풍부할수록 planner가 더 정교한 문서 계획을 생성할 수 있어야 한다
* 목표 길이는 short, medium, long 수준으로 받되 내부적으로 섹션별 목표 분량으로 변환해야 한다

---

## 8.2 Planner

Planner는 문서 설계의 중심이다.

### 책임

* 사용자 요청 해석
* 문서 목적 정리
* 독자 수준 정리
* 문서 형식 판별
* 전체 아웃라인 설계
* 섹션별 작성 브리프 생성
* 병렬 가능 여부 및 의존성 결정

### 출력

Planner는 아래 정보를 구조화된 형태로 출력해야 한다.

* 전역 문서 브리프
* 문서 제목
* 전체 아웃라인
* 각 섹션의 목표
* 각 섹션의 핵심 포인트
* 각 섹션의 최소 분량
* 각 섹션의 의존성
* 각 섹션의 병렬 가능 여부
* 중복 방지 규칙

### 요구사항

* 아웃라인은 최소 5개, 최대 9개 섹션을 기본 범위로 한다
* 각 섹션은 관찰 가능한 산출물을 만들어야 한다
* 각 섹션은 가능한 한 하나의 명확한 목적을 가져야 한다
* planner는 추상적 섹션만 나열하지 말고 section brief까지 생성해야 한다

---

## 8.3 Section Executor

Section Executor는 각 섹션 초안을 작성한다.

### 책임

* 전역 브리프와 섹션 브리프를 입력으로 받는다
* 섹션 하나의 초안을 작성한다
* 최소 분량 기준을 만족해야 한다
* 지정된 예시, 코드, 설명 요소를 포함한다
* 중복 회피 규칙을 따른다

### 요구사항

* 각 섹션은 독립적으로 작성 가능해야 한다
* 섹션 목표에 따라 설명, 예시, 비교, 코드 블록 등을 포함할 수 있어야 한다
* 섹션 단위 실패가 전체 문서 생성 실패로 이어지지 않도록 재시도 가능해야 한다
* 섹션은 draft 상태와 review 상태를 구분해야 한다

---

## 8.4 병렬 실행

문서 에이전트는 제한적 병렬 실행을 지원해야 한다.

### 병렬 실행 대상

* 의존성이 없는 섹션 초안 작성
* 예시 추가
* FAQ 추가
* 분량 보강 작업

### 병렬 실행 비대상

* 최초 아웃라인 생성
* 문서 전체 흐름 설계
* 서론/결론의 최종 작성
* 최종 편집 및 확정

### 요구사항

* 시스템은 planner가 정의한 dependency를 기준으로 병렬 그룹을 구성해야 한다
* 병렬 실행 결과는 이후 편집 단계에서 통합 가능해야 한다
* 병렬로 작성된 섹션 간 용어 및 내용 중복은 editor 단계에서 해결해야 한다

---

## 8.5 Editor

Editor는 생성된 여러 초안을 하나의 완성된 문서로 통합한다.

### 책임

* 섹션 병합
* 용어 통일
* 중복 제거
* 섹션 간 연결 문장 추가
* 전역 톤 맞춤
* 과도한 요약 방지

### 요구사항

* 편집 후 문서가 원본 초안보다 과하게 짧아지지 않아야 한다
* 서론, 본문, 결론의 연결성이 개선되어야 한다
* 중복 설명이 줄어들어야 한다
* 전역 스타일 가이드를 반영해야 한다

---

## 8.6 Reviewer

Reviewer는 문서 품질을 점검하고 보강 필요 영역을 식별한다.

### 책임

* 필수 섹션 포함 여부 검사
* 섹션별 최소 분량 충족 여부 확인
* 논리 흐름 점검
* 독자 적합성 점검
* 중복 및 누락 확인
* 약한 섹션 식별

### 요구사항

* reviewer는 pass/fail만 내리지 않고 구체적인 수정 포인트를 반환해야 한다
* 문제가 있는 섹션을 식별하고 patch 대상 목록을 제공해야 한다
* reviewer 기준은 체크리스트 기반으로 구조화되어야 한다

---

## 8.7 Patch / Rewrite

Patch 단계는 reviewer가 식별한 약한 섹션만 보강한다.

### 책임

* 짧은 섹션 확장
* 부족한 예시 추가
* 모호한 설명 보강
* 누락된 하위 포인트 추가

### 요구사항

* 전체 문서를 다시 생성하지 않고 필요한 섹션만 수정할 수 있어야 한다
* 보강 후 editor 또는 reviewer를 다시 호출할 수 있어야 한다
* patch는 최대 1~2회 반복을 기본값으로 둔다

---

## 9. 상태 모델 요구사항

시스템은 문서 작성 상태를 추적할 수 있어야 한다.

### 전역 상태

* request
* global brief
* outline
* merged draft
* final document
* logs

### 섹션 상태

* todo
* ready
* writing
* written
* editing
* done
* failed

### 요구사항

* 각 섹션의 상태를 개별 추적해야 한다
* 실패 섹션을 재실행할 수 있어야 한다
* 로그를 통해 어느 단계에서 품질 문제가 발생했는지 확인할 수 있어야 한다

## 10. 사용자 스토리

### US-1

사용자로서, 나는 긴 기술 문서를 작성할 때 구조가 잘 잡힌 아웃라인을 먼저 얻고 싶다. 그래야 본문 품질이 흔들리지 않는다.

### US-2

사용자로서, 나는 각 섹션이 충분히 길고 깊이 있게 작성되길 원한다. 그래야 단순 요약이 아닌 실제 사용 가능한 문서를 얻을 수 있다.

### US-3

사용자로서, 나는 문서 작성 속도를 높이기 위해 독립적인 섹션들은 병렬로 생성되길 원한다. 그래야 전체 생성 시간이 줄어든다.

### US-4

사용자로서, 나는 생성된 문서의 특정 섹션만 다시 쓰거나 보강하고 싶다. 그래야 전체를 다시 생성하지 않아도 된다.

### US-5

사용자로서, 나는 최종 문서가 하나의 사람이 쓴 것처럼 자연스럽고 일관되길 원한다. 그래야 실제 문서로 바로 활용할 수 있다.

---

## 11. 비기능 요구사항

### 성능

* 병렬 가능한 섹션은 동시 실행할 수 있어야 한다
* 전체 생성 시간은 동일한 문서를 순차적으로 전부 생성하는 방식보다 개선되어야 한다

### 품질

* 단일 호출 방식 대비 평균 문서 길이가 유의미하게 증가해야 한다
* 아웃라인 completeness가 향상되어야 한다
* 섹션별 최소 분량 기준을 만족해야 한다

### 확장성

* 새로운 executor 유형을 쉽게 추가할 수 있어야 한다
* 문서 형식별 planner 규칙을 확장할 수 있어야 한다
* 추후 검색, 인용, 데이터 기반 작성 기능을 붙일 수 있어야 한다

### 관측성

* 단계별 로그와 섹션 상태를 추적할 수 있어야 한다
* planner 출력과 reviewer 결과를 저장할 수 있어야 한다

## 13. 성공 지표

### 핵심 지표

1. **아웃라인 완성도**

   * 필수 섹션 포함률
   * 중복 섹션 감소율
   * reviewer 기준 통과율

2. **문서 깊이**

   * 단일 호출 대비 평균 단어 수 증가
   * 섹션별 최소 분량 충족률
   * 예시/세부 설명 포함률

3. **문서 일관성**

   * editor 후 중복 감소율
   * 용어 불일치 감소율
   * reviewer 품질 점수 향상

4. **생산성**

   * 부분 병렬 적용 시 전체 생성 시간 감소
   * 사용자의 수동 수정 횟수 감소

### 초기 KPI 예시

* 단일 호출 대비 평균 문서 길이 1.5배 이상
* reviewer pass rate 70% 이상
* 필수 섹션 누락률 10% 이하
* 병렬 가능한 문서에서 생성 시간 20% 이상 개선

## 14. 리스크와 대응

### 리스크 1: 병렬 섹션 간 중복 증가

대응:

* planner가 avoidOverlapWith 필드를 생성
* editor에서 중복 제거 전용 편집 수행

### 리스크 2: 용어 불일치

대응:

* global brief에 glossary와 style guide 포함
* editor에서 용어 통일 수행

### 리스크 3: 일부 섹션만 지나치게 약함

대응:

* reviewer가 weakSections를 반환
* patch 단계에서 부분 보강

### 리스크 4: planner 품질이 낮으면 전체 문서 품질이 흔들림

대응:

* planner output schema를 강하게 구조화
* outline에 대한 sanity check 추가

### 리스크 5: editor가 문서를 과도하게 압축

대응:

* 편집 프롬프트에 길이 유지 원칙 명시
* reviewer에서 길이 후퇴 검증

## 15. 출시 기준

초기 버전은 아래 조건을 만족하면 출시 가능하다.

* planner가 구조화된 outline과 section brief를 안정적으로 생성한다
* 의존성 없는 섹션 병렬 생성이 가능하다
* editor가 초안들을 하나의 문서로 자연스럽게 통합한다
* reviewer가 약한 섹션을 식별할 수 있다
* patch 단계로 특정 섹션 보강이 가능하다
* 단일 호출 대비 길이와 구조 품질 개선이 확인된다

## 17. 최종 요약

이 제품은 문서 작성을 단일 생성이 아니라 **계획 → 섹션별 실행 → 편집 → 검토 → 보강**의 구조적 워크플로우로 바꾸는 것을 목표로 한다.

핵심 차별점은 세 가지다.

* 전역 planner가 먼저 고품질 아웃라인과 섹션 브리프를 만든다
* executor가 섹션별로 충분한 깊이의 초안을 작성한다
* editor와 reviewer가 문서를 하나의 완성된 산출물로 만든다

그 결과, 기존 단일 LLM 호출에서 자주 발생하던 **짧은 문서, 부실한 아웃라인, 낮은 일관성** 문제를 체계적으로 완화할 수 있다.
